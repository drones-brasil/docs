## Objetivo
Entender o ecossistema PX4 + ROS 2 e implementar uma **classe Drone** que encapsula comunicação e comandos (getters/setters).

## 1. PX4 Autopilot (visão geral)
O **PX4** é um firmware open-source que funciona como “SO do drone”, cuidando de:
- estabilização/atitude
- fusão de sensores
- modos de voo (manual, auto, offboard)
- interface com sistemas externos (MAVLink e MicroXRCE-DDS)

## 3. px4_msgs e o Agente DDS

### 3.1 O que é `px4_msgs`
Mensagens ROS 2 do PX4 (seguem uORB).

Listar tópicos:

```bash
source install/setup.bash
ros2 topic list
```

Mostrar interface de uma msg:

```bash
ros2 interface show px4_msgs/msg/VehicleOdometry
```

### 3.2 MicroXRCE-DDS
Ponte PX4 ↔ ROS 2.

```bash
micro-xrce-dds-agent udp4 -p 8888
```

## 4. Criando a classe Drone (conceito)

### 4.1 Objetivo da classe
- Subscribers (ex.: `/fmu/out/vehicle_status`, `/fmu/out/vehicle_odometry`)
- Getters (ex.: `getArmingState()`, `getLocalPosition()`, `getOrientation()`)
- Publishers (ex.: `/fmu/in/vehicle_command`, `/fmu/in/trajectory_setpoint`)
- Setters (ex.: `toOffboardSync()`, `setLocalPosition(...)`, `armSync()`)

### 4.2 Headers vs source (.hpp vs .cpp)
Separar interface e implementação para build mais eficiente.

## 6. Publishers e Setters (exemplo)

### 6.1 O que é Offboard (intuição)
Offboard só “engata” se estiver recebendo setpoints a uma taxa mínima:
- `/fmu/in/offboard_control_mode`
- `/fmu/in/trajectory_setpoint`

### 6.2 Setters principais
- `sendCommand(...)`
- `setOffboardMode()`
- `arm()` / `disarm()`
- `setOffboardControlMode(...)`
- `setLocalPosition(...)`
- `toOffboardSync()` (publica setpoints antes de trocar o modo)
